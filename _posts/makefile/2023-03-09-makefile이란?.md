## **_Makefile?_**

프로그램 개발 시 한 파일에서 모든 것들을 개발할 수 없으므로, 여러 개의 모듈로 나누어서 개발을 진행하게 된다. 이 때,
입력 파일이 바뀌게 되면 바뀐 파일과 관계가 있는 파일 또한 다시 컴파일 해야 하는 불편함이 발생한다. 이 때, Makefile을
만들어 빌드시 불편함을 줄일 수 있다.

즉, `특정 프로그램의 일부분이 수정돼 재 컴파일이 필요한 경우` 용이하게 사용할 수 있는 프로그램이다.

<hr>

## **_왜 Makefile이 필요할까?_**

### **_간편성_**

프로그램 개발의 규모가 커질수록 의존성을 모두 체크하기가 어렵고, 일일이 하나씩 컴파일해서 링킹까지... 너무 어렵다.
컴퓨터 공학에 오면서 어떤 문제가 심각한지 이해하기 위해서는 필요한 횟수를 엄청나게 큰 수로 늘려서 생각해보라고 들었다.
만약에 컴파일해야 하는 파일이 1만 개 이상이라고 생각해보자. 또한 의존성이 어떻게 되어 있는지 모른다면?? 하나하나
찾아보기에는 너무나 노가다성의 힘든 작업이 될 것이다. 예를 들어, Makefile을 사용할 때와 사용하지 않을 때
입력해야 하는 명령어를 보면 아래와 같다.

`Make 사용하지 않을 때`

```bash
# object file 생성
gcc -c -o main.o main.c
gcc -c -o file1.o file1.c
gcc -c -o file2.o file2.c
gcc -c -o file3.o file3.c
# ...
gcc -c -o file10000.o file10000.c

# linking
gcc -o a.out main.o file1.o file2.o ... file10000.o

# 실행
./a.out
```

`Make 사용하고 있을 때`

```bash
# 스크립트에 넣어서 빌드
make

# 실행
./a.out
```

누가봐도 아래의 Makefile을 사용하는 것이 더 간단하고 편하다고 생각할 것이다. 또한 파일이 추가될 때 혹은 수정될 때마다
같은 작업을 할 때, 더 효과가 빛날 것이다.

### **_안전성_**

중요한 사실은 간단하고 편한 것 이외에도 `안전의 이유로 Makefile을 사용하는 것이 좋다.` 아래의 예시를 생각해보자.

```bash
gcc -c main.c
```

이렇게 object file을 생성할 때, .o가 아닌 .c로 실수로 작성하면 어떤 일이 벌어질까? 👀 눈알이 이렇게 튀어나올지도
모른다. 왜? 지금까지 작업한 main.c 파일(소스파일)이 바이너리 파일로 변환되면서 사라지기 때문이다! Makefile을
사용한다면 파일에 작성되어 한 번 더 확인해볼 수 있지만 커맨드에 바로 오브젝트 파일을 만들도록 하면 이러한 치명적인
실수가 나타나면 돌이킬 수 없게 된다.

### **_의존성_**

뿐만 아니라, 파일마다의 의존성을 전부 다 외우고 있어야 하는데 이는 파일의 개수가 많아질 수록 더욱 더 힘들어질 것이다.
Makefile에서는 `Incremental build`라는 기능이 있어 빌드 과정에서 변경된 소스코드에 의존성이 있는 대상들만
추려서 다시 빌드하게 된다. 즉, main.c 소스파일을 수정했을 경우 main.o 컴파일와 a.out 링크만 다시 수행하게 된다.

`그냥 깔끔하게 전부 다시 빌드하면 안돼? 쉘 스크립트를 만들어서?`

이렇게 생각할 수도 있는데, 대규모 프로젝트의 경우 빌드 대상 소스파일의 개수와 복잡도가 엄청나게 클 뿐더러 모두 컴파일부터
링킹까지 엄청난 시간이 소요될 것이다.

<hr>

이외에도 다양한 기능들이 있지만, 이 정도만 생각해도 Makefile을 사용하는 편이 더 효율적이라는 것을 알 수 있을 것이다.
그렇다면 Makefile은 어떻게 만드는 지에 대해서는 다음에 포스팅하도록 하자...
